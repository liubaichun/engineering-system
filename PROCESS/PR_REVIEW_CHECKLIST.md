# Pull Request 审核清单

## 审核要求

- 每次PR必须经过至少1人审核才能合并
- 审核人应该是代码所有者或有相关模块经验的开发者
- 作者不能审核自己的PR

## 审核检查项

### 1. 代码质量
- [ ] 代码逻辑正确，没有明显的bug
- [ ] 异常处理完善
- [ ] 没有硬编码密码/密钥/Token
- [ ] 没有临时调试代码（console.log, print等）
- [ ] 符合代码规范

### 2. 功能完整性
- [ ] 实现了需求文档中的功能
- [ ] API接口设计合理
- [ ] 前端交互流畅
- [ ] 有必要的表单验证

### 3. 安全性
- [ ] 用户输入有校验
- [ ] SQL注入防护
- [ ] XSS防护
- [ ] 权限校验正确
- [ ] 敏感信息不暴露

### 4. 数据库变更
- [ ] 新增字段有默认值或允许null
- [ ] 不删除已有字段（用deprecated替代）
- [ ] 索引合理
- [ ] 有migration脚本

### 5. 前后端一致性
- [ ] Serializer字段名与前端表单字段名一致
- [ ] API响应包含前端需要的显示字段
- [ ] 新增字段已同步更新前后端

### 6. 测试
- [ ] 单元测试覆盖（如适用）
- [ ] 测试用例合理
- [ ] 不破坏现有功能

### 7. 文档
- [ ] API变更有文档更新
- [ ] 有必要的注释说明复杂逻辑

## 审核结论

| 结论 | 含义 | 后续动作 |
|------|------|----------|
| ✅ 通过 | 可以合并 | 审核人批准，通知作者 |
| ❌ 打回 | 需要修改 | 列出具体问题，作者修改后重新提交 |
| ⚠️ 有条件通过 | 需记录注意事项 | 记录并通知作者，可合并但需后续跟进 |

## 打回格式

```
❌ 打回原因：xxx

需要修改：
1. xxx
2. xxx

请修改后重新提交审核。
```

## 通过格式

```
✅ 审核通过

审核内容：xxx
变更内容：xxx
注意事项：xxx（如有）

下一步：请合并PR
```

## 合并规则

1. 所有CI检查必须通过
2. 至少1人审核通过
3. 无冲突
4. 分支保护规则必须满足
